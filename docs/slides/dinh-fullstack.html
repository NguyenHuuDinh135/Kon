4. AI Agent — LangGraph Architecture

packages/ai-engine/ai_engine/agent.py
# ReAct Pattern: Reason → Act → Observe → Repeat

from langgraph.graph import StateGraph, END
from langgraph.prebuilt import ToolNode
from langchain_aws import ChatBedrock

# LLM selection (Bedrock primary, Ollama fallback)
if USE_BEDROCK:
    llm = ChatBedrock(
        model_id="us.anthropic.claude-haiku-4-5-20251001-v1:0",
        region_name="us-west-2"
    )
else:
    llm = ChatOllama(model="qwen2.5", base_url=OLLAMA_URL)

# Bind 7 tools
llm_with_tools = llm.bind_tools(tools)

# Graph: agent_node → should_continue? → tool_node → agent_node → ...
graph = StateGraph(AgentState)
graph.add_node("agent", call_model)
graph.add_node("tools", ToolNode(tools))
graph.add_conditional_edges("agent", should_continue, {
    "continue": "tools",
    "end": END
})
graph.add_edge("tools", "agent")

# Two modes:
# run_agent(prompt) → single response
# stream_agent(prompt) → SSE events for real-time UI

8. Database Design Decisions

# 3-layer data topology:
#
# Layer 1: Olist (ERP Core) — 9 tables
#   Real Brazilian e-commerce data, multi-marketplace
#   Used for: revenue analysis, product recommendations, geographic insights
#
# Layer 2: Online Retail — 1 table (541K transactions)
#   UK online retail transactions
#   Used for: RFM calculation → K-Means clustering
#   Why separate: different granularity (transaction-level vs order-level)
#
# Layer 3: E-Commerce Churn — 1 table (5,630 customers)
#   Supervised labels (real Churn = 0/1)
#   Used for: Decision Tree + Logistic Regression training
#   Has pgvector embedding column (1024-dim) for semantic search
#
# System tables: users, audit_logs, notifications, campaigns, system_alerts
# ML tables: ml_model_metrics (versioned), ml_recommendations, customer_segments

# pgvector usage:
# - Column: customer_churn.embedding (Vector(1024))
# - Index: HNSW for approximate nearest neighbor
# - Operator: <=> (cosine distance)
# - Use case: "find customers similar to this description"

9. API Design — Key Patterns

# Authentication flow:
# POST /auth/login → JWT token (24h expiry)
# All protected routes: Depends(get_current_user)
# Admin routes: Depends(require_admin)

# Rate limiting (slowapi):
# - Auth endpoints: 5/min (prevent brute force)
# - Search: 30/min
# - General: 60/min

# Response patterns:
# Success: {"data": [...], "count": 50}
# Error: HTTPException(status_code=4xx, detail="message")
# Streaming: StreamingResponse for agent chat (SSE)

# SSR optimization:
# - INTERNAL_API_URL=http://api:8000 (container-to-container, no DNS hop)
# - NEXT_PUBLIC_API_URL=http://localhost:8000 (client-side browser calls)
# - Server components fetch via INTERNAL_API_URL
# - Client components fetch via NEXT_PUBLIC_API_URL

11. Kiến thức cần nắm vững để bảo vệ

Câu hỏi thầy có thể hỏi:
  1. Tại sao chọn 3 datasets riêng biệt thay vì 1 dataset lớn?
    → Mỗi dataset phục vụ mục đích ML khác nhau (unsupervised clustering vs supervised classification vs time series). Kết hợp tạo ra hệ thống phong phú hơn.
  2. Tại sao dùng pgvector thay vì Elasticsearch?
    → pgvector tích hợp trực tiếp vào PostgreSQL, không cần service riêng. Phù hợp scale 5K-10K vectors. Elasticsearch overkill cho use case này.
  3. AI Agent hoạt động thế nào?
    → ReAct pattern: nhận câu hỏi → reasoning chọn tool → gọi tool → đọc kết quả → reasoning tiếp → trả lời. Loop tối đa 5 iterations.
  4. Tại sao retrain mỗi 4 giờ?
    → Balance giữa freshness và resource cost. Data thay đổi chậm (batch ETL), 4h đủ để capture trends mới mà không waste compute.
  5. Docker Compose vs Kubernetes?
    → Scope dự án là local/demo, 4 services đơn giản. K8s overkill. Docker Compose đủ cho development + demo.
  6. Bedrock vs Ollama?
    → Bedrock cho production (Claude Haiku — fast, accurate, Vietnamese). Ollama cho offline/demo khi không có internet. Dual-provider strategy.
